# vi

## 배우는 이유

- 시리얼 포트로 코드를 수정할 때 vi로 편집할 수 있다.
- 정규표현식을 만든 해커들이 좋아하는 vi 편집기
- 쓸만하다면 메인 편집기를 vi로 사용하면 좋다
- 생산성 최상

## Basic

- `vim 이미 있는 파일`  → 파일이 열림
    
    `vim 없는 파일` → 해당 이름을 갖은 파일이 생성되면서 열림
    
- 명령 모드 ↔ 입력 모드 가능
    - 명령 모드 → 입력 모드 : i 키
    - 입력 모드 → 명령 모드 : Esc 키
- 명령모드 ↔ 실행 모드 가능
    - 명령 모드 → 실행 모드 : `:` 키
    - 실행 모드 → 명령 모드 : Esc 키
- but, 입력모드 ↔ 실행 모드 불가능
- vim 편집기에서 커서를 이동할 때 - 방향키도 가능하지만 H, J, K, L 키로 이동하는 것이 좋음
    - H : 좌
    - J : 하
    - K : 상
    - L : 우
- `:set nu`  → 줄번호 나오게 만들기
- 명령 모드에서 `ddp` 입력하면 커서가 있는 줄과 아래줄 스왑
    - `dd` 는 잘라내가 `p`는 붙여 넣기
- 명령 모드에서 `u` 입력하면 이전상태로 되돌리기
- 명령 모드에서 `6dd` 를 사용하면 6줄 한 번에 지우기 가능
- 명령 모드에서 `/` → 실행 모드에서 검색모드
    - 다음 `n`
    - 이전 `shift + n`
- 명령 모드에서 `:%s/변경할 것/새롭게 변경할 것` 치환
    - 명령 모드에서 `:%s/변경할 것/새롭게 변경할 것/c` → 순서대로 치환할 것인지 아닌지 물어봄
- vim 종료
    - `:q!` : 저장 안하고 종료
    - `:w` : 저장
    - `:wq`  : 저장 후 종료
    - `shift + zz` : 저장 후 종료
- 다른 파일과 현재 파일 병합시키기 → `:sp  병합할 파일명` → 현재 파일과 병합할 파일이 상하로 열림
    
    `v` : 비주얼 모드 → 커서만 움직여도 블록으로 선택 처리됨 → 복사 `y` (yank)
    
    → 커서를 현재 파일의 vim으로 이동 `ctrl + ww` → 붙여 넣기 `p` 
    
    → 위의 창 닫기 `위로 커서 올리고 shift + zz` 
    
- 커서가 위치한 한글자 지우기 `x`
- 커서를 맨 뒤으로 옮기기 `shift + 4`  = `$`
- 커서를 맨 앞으로 옮기기 `shift + 6`  = `^` , `0`
- 한 줄을 복사할 때 `yy`
- 한 번에 여러번 붙여 넣기 ex 5번 붙여 넣기 `5p`
- 특수 문자 검색 치환 `:%s/\$/\&` 이런 식으로 `\` 이스케이프 시켜줘야 함
- 특정 줄로 이동ex 36번째 줄 → `36G`
- 한 줄 띄우기 → `shift + o`

# 리눅스 가상 메모리

## 메모리란

컴퓨터가 동작하기 위해서는 메모리(memory) 가 필요하다.

메모리는 데이터를 임시로 저장하고, CPU가 이를 읽거나 쓸 수 있도록 해주는 공간이다.

모든 프로그램이 동시에 충분한 물리 메모리를 사용할 수 있다면 좋겠지만, 현실적으로 이는 어렵다.

이때 운영체제는 제한된 메모리를 효율적으로 활용하기 위해 메모리 관리(Memory Management) 를 수행한다.

리눅스 역시 메모리 관리를 통해 다음과 같은 중요한 기능을 수행한다.

- 한정된 메모리를 프로세스들에게 공정하게 나누어준다.
- 프로세스 간의 메모리를 보호하여 안정성을 확보한다.
- 메모리의 낭비를 방지한다.

## 물리 VS 가상 메모리

### 물리 메모리

- 컴퓨터가 실제로 가지고 있는 RAM을 의미한다.
- 프로그램이 실제 데이터를 저장하는 장소로, 용량이 제한적이다.

### 가상 메모리

- 프로그램이 사용하는 메모리를 물리 메모리보다 크거나 효율적으로 관리하기 위해 운영체제가 제공하는추상화된 메모리 주소 공간
- 프로세스는 가상 메모리 주소만 알고 있고, 물리 메모리는 운영체제가 관리한다.
- 프로세스마다 독립된 가상 주소 공간을 갖기 때문에 보안과 안정성이 높아진다.
- 임베디드 리눅스에서 GPIO 레지스터 주소 조차 가상 메모리의 주소이다

| 특징 | 물리 메모리 | 가상 메모리 |
| --- | --- | --- |
| 위치 | 실제 하드웨어 RAM | 논리적 주소 공간 |
| 주소타입 | 물리 주소 | 가상 주소 |
| 관리 주체 | OS 커널 | 운영체제의 소프트웨어 + 하드웨어 지원(MMU)  |
| 프로세스 관점 | 직접 접근 불가 | 직접 접근 가능 |

### 가상 메모리 사용 이유

- 멀티태스킹:
    - 여러 프로세스가 독립된 가상 주소 공간을 가져, 서로 영향을 미치지 않고 실행 가능.
- 메모리 보호(Memory Protection):
    - 프로세스가 서로 다른 주소 공간에서 동작하기 때문에 다른 프로세스의 데이터를 보호한다.
- 효율적인 메모리 관리
    - 메모리를 효율적으로 관리할 수 있도록 프로세스가 필요한 부분만 로드하여 메모리 사용량을 절
    약할 수 있다.
- 메모리 단편화 방지
    - 물리 메모리의 연속적인 공간을 확보할 필요가 없어서 메모리의 작은 공간을 모아서 효율적으로
    사용할 수 있다.

### 가상 메모리의 기본 작동 방식 개요

가상 메모리 시스템은 일반적으로 **페이지(page)**라는 고정된 크기의 단위로 메모리 관리

- 가상 주소 : 프로세스가 사용하는 주소
- 물리 주소 : 실제 RAM이 사용하는 주소
- 페이지 : 가상 주소의 단위
- 페이지 프레임 : 물리 메모리 단위

### 주소 변환 예시

프로세스(가상 주소) ----[MMU 주소변환]----> 물리 주소

MMU는 V-address를 P-address로 변환해주는 기능 (반대는 불가능)

이 변환 과정에서 리눅스는 **페이지 테이블**을 사용하여 가상 주소를 물리 주소로 매핑

## 가상 주소와 물리 주소의 변환 과정

### 주소 변환

[프로세스] --> 가상 주소 --> [MMU] --> 물리 주소 --> [RAM]

### 가상 주소의 구조 이해

32비트 시스템의 가상 주소 구성

```yaml
| 페이지 번호(Page Number) | 페이지 내 오프셋(Page Offset) |
```

- 페이지 번호 : 어떤 페이지인지 나타내는 부분
- 오프셋 : 해당 페이지 내의 위치를 나타내는 부분

### MMU와 페이지 테이블의 역할

- MMU
    - CPU가 사용하는 가상 주소를 실제 주소로 변환하는 하드웨어
    - MMU는 페이지 테이블을 통해 변환 정보 얻음
- 페이지 테이블
    - 가상 페이지 번호와 물리 페이지 프레임의 관계를 기록한 표
    - 운영체제의 커널이 관리
    - 페이지 테이블은 크기가 매우 크기 때문에, 성능 향상을 위해 MMU는 **TLB**라는 고속 캐시를 활용

## Make란

**Make는 프로그램 빌드 과정을 자동화하는 도구이다.**

소스 코드가 많아질수록 컴파일 → 링크 → 실행파일 생성 과정을 수동으로 관리하기 어려워지는데 Make는 이를 **Makefile에 정의된 규칙으로 자동 처리한다.** 
Make 도구 특히 **수정된 파일만 다시 컴파일**하여 빌드 시간을 줄인다.

빌드 과정

```
source code → compile → link → executable
```

## Makefile 기본 구조

Make는 **Makefile에 정의된 규칙(rule**)을 읽어 빌드를 수행한다.

기본 문법

```
target: dependency
    command
```

구성 요소

| 요소 | 의미 |
| --- | --- |
| target | 생성할 결과 파일 |
| dependency | 타겟을 만들기 위해 필요한 파일 |
| command | 실행할 명령 |

예시

```
main: main.c hello.c
    gcc main.c hello.c -o main
```

→ `make main` 실행 시 프로그램 생성. 
Make 도구

## Dependency (의존성)

Make의 가장 중요한 개념이다.

예

```
main: main.o hello.o
```

의미

```
main 실행파일 생성
→ main.o 필요
→ hello.o 필요
```

그리고

```
main.o → main.c
hello.o → hello.c
```

즉

```
main.c 변경 → main.o 재컴파일
hello.c 변경 → hello.o 재컴파일
```

이렇게 **변경된 파일만 다시 빌드**한다.

## Makefile 변수

Makefile에서는 **명령어 반복을 줄이기 위해 변수 사용**이 중요하다.

예

```
CC = gcc
CFLAGS = -Wall -g

program: main.c
    $(CC) $(CFLAGS) main.c -o program
```

장점

- 컴파일러 변경 쉬움
- 옵션 관리 쉬움

## Pattern Rule (패턴 규칙)

여러 파일에 동일한 규칙을 적용할 때 사용한다.

예

```
%.o: %.c
    gcc -c $< -o $@
```

의미

```
모든 .c 파일 → .o 파일 생성
```

여기서 자동 변수 사용

| 변수 | 의미 |
| --- | --- |
| `$@` | target |
| `$<` | 첫 dependency |
| `$^` | 모든 dependency |

## .PHONY 타겟

실제 파일이 아닌 **명령 실행용 타겟**

예

```
.PHONY: clean

clean:
    rm -f *.o program
```

사용

```
make clean
```

→ 빌드 파일 삭제

## Make 주요 기능

- 병렬 빌드

```
make -j4
```

→ CPU 코어 수만큼 동시에 빌드

- 쉘 명령 실행

```
$(shell date)
```

→ Makefile에서 쉘 명령 실행

- 와일드카드

```
$(wildcard *.c)
```

→ 모든 C 파일 자동 검색

## 자주 사용하는 타겟

| 타겟 | 역할 |
| --- | --- |
| all | 기본 빌드 |
| clean | 빌드 파일 삭제 |
| install | 프로그램 설치 |
| dist | 배포 패키지 생성 |

## Make를 사용하는 이유

Make는 다음 문제를 해결한다.

- 소스 파일이 많아질 때 빌드 관리 어려움
- 의존성 관리 실수
- 전체 재컴파일로 인한 시간 낭비

해결

```
자동 의존성 관리
부분 컴파일
빌드 자동화
```
